## Sequencing (per-ticket dispatch — the P1.1/P2 default)

Per-ticket dispatch (protocol § "Token economy"): **one ticket per Developer subagent,
in dependency order, each ending on a real checkpoint.** These four share the movement
spec as a read but each lands a distinct, separately-verifiable checkpoint and one
(TKT-…-03) carries an engine change, so **no cluster is warranted** — the spec-read
overlap does not outweigh the blast-radius separation.

```
01 trace-harness ─▶ 02 geometry-Y-fix ─▶ 03 held-input-stances ─▶ 04 airborne-actions
   (the instrument     (AD-037 box data      (AD-038 walk exit +      (AD-039 dir/diag
    the rest verify      reflection;           crouch stance/block)     jumps + air
    through)             render is correct)                             normals)
```

Dispatch order and why:

1. **TKT-P1.1R-01 (trace harness)** first — it is the **verification instrument** the
   work-order folds in. Building it first lets the Developer verify each movement fix
   below **through it** as they land (the brief checklist becomes assertions). It is
   blind to render bugs (documented), so it shrinks but does not remove the human gate.
   *Checkpoint:* a string compiles to a buffer, replays headless through
   `RecordPlaybackSource`, and a smoke assertion (e.g. "idle stays idle") passes.
2. **TKT-P1.1R-02 (geometry Y-fix)** next — **AD-037** is the single root cause of the
   whole gate-2 box-appearance cluster, and fixing it unblocks the visual re-gate. It is
   a **data** change (reflect every authored box) independent of the movement *logic*
   below, so it lands cleanly early. *Checkpoint:* boxes reflected; geometry goldens
   re-baselined; combat/advantage goldens **unchanged**; harness/sim shows the pushbox at
   the body's lower part and the crouch box shrinking downward.
3. **TKT-P1.1R-03 (held-input stances)** — the **AD-038** engine change (phase-2 loop-state
   exit) plus the crouch-stance wiring; walk now stops on release and crouch enters/exits.
   Carries the only engine edit, so it is isolated. *Checkpoint:* harness asserts walk-and-
   stop both ways and crouch enter-on-hold / stand-on-release.
4. **TKT-P1.1R-04 (airborne actions)** — the **AD-039** content wiring: forward/back &
   diagonal jumps and air-normal reachability. Pure `content/character_a.gd`. Last because
   it is the most content and benefits from the harness + a settled state machine.
   *Checkpoint:* harness asserts `JUMP_F/JUMP_B` reachable on `9/7`, and `j.L/M/H`
   reachable in the air.

**The dash is NOT a ticket.** The brief's *discretionary* forward/back dash is a
Strategist/user **scope** call, raised as a flag (`flags.md`, owner: Strategist) — the
dash **states** exist and are correct data, but the input to reach them (double-tap `66`/
`44`) is unbuilt and would need a **new** recognizer mechanism. Do not build it under this
file until the scope flag resolves.

**After all four:** QA runs the objective audit — the movement/geometry changes **move sim
behavior and box geometry deliberately** (goldens change, JC-017 style; the *relationships*
the combat goldens test are unchanged by the uniform box reflection). Then the **user
re-gate** drives the brief checklist (`character-a-movement-reconciliation.md` →
"Acceptance"). QA cannot mark P1.1 done while the gate is open; only the user closes it.

**Deferred / parked (confirm at the re-gate, do not lose):** the two feel flags
(frame-step auto-pause JC-045; jump apex-hang JC-047) and the crouching-normal-height flag
(likely closes once AD-037 lands — re-evaluate against a correctly-oriented display, don't
close it before). All non-blocking; all Strategist's to close.

### TKT-P1.1R-01 · Scripted-input behavioral-trace harness (minimal)
**Serves:** `/docs/spec/trace-harness.md` (all three contracts + acceptance 1–6); Tenet 2;
AD-011 (reads only through `InspectionView`); AD-019 (float-free trace). **Depends:**
nothing new — `RecordPlaybackSource` (JC-030), `InspectionView`, `MoveRegistry`, the
`TrainingHarness`/`TickHost` produce-before-query wiring all exist.
**Scope (minimal — hold the line):**

- **`InputScript` compiler** (`compile(text) -> PackedInt32Array`, a pure `class_name` of
  static helpers, sibling to `InputFrame`): the numpad+`L/M/H`+`*count`+`#`-comment grammar
  in the spec, RAW/screen-relative numpad, per-tick motion authoring, hard-error on a
  malformed token or reserved bit (validated at the boundary, `input.md` criterion 6).
- **Headless driver**: compile P1 (and optional P2, default neutral) → buffer(s); load each
  into a `RecordPlaybackSource` via `set_recorded_buffer` + `Mode.PLAYBACK`; install the
  roster; build the initial `SimState` with both players as the character under test in idle
  (same wiring TKT-P1.1-01 fixed); run N ticks with `produce_next()`-then-`step`.
- **Trace dump + inline-assert runner**: build a trace row per tick from `InspectionView`
  (the default movement field-set; optional `boxes`/`advantage`/`last_hit`), fixed field
  order, **no floats**; support inline `(tick, field, expected)` assertions that fail loudly
  with tick/field/expected/actual.
- **Not now:** P2 mirroring, `.trace` golden tooling, dummy AI, any non-`RecordPlaybackSource`
  replay path.

**Acceptance:** `trace-harness.md` criteria 1–6. Include a smoke script (idle stays idle;
`6*n` moves `pos_x` forward) proving compile→replay→assert end-to-end and source-equivalent /
deterministic replay (criterion 3). **Judgment-log:** record any latitude (grammar edge
cases, the trace row exact text) for ratification.

### TKT-P1.1R-02 · Geometry Y-fix — box-data reflection (AD-037)
**Serves:** **AD-037**; `move-format.md` → `Box` (vertical convention); `training-mode.md`
criteria 5/14 (boxes render right-side-up). **Depends:** TKT-P1.1R-01 (to assert sim box
orientation through the harness; render itself is the human gate). Independent of the
movement-logic tickets.
**Scope:**

- **Reflect every authored box across the feet line** (`new_y = −(old_y + old_h)`, `h`
  unchanged) in `content/character_a.gd` — hurtboxes (`_hurt_stand`/`_hurt_crouch`/
  `_hurt_air`), the default pushbox, every normal/special/throw hitbox, and every spawn
  offset that is a vertical body reference. **Do the same for the P0 test character**
  (`tests/test_support.gd`) so the one box convention holds slice-wide (AD-037).
- **No engine/format code change.** `resolve_box`, `ResolvedBox.overlaps`, and `px_rect`
  already assume the shared-axis translate (up = −Y); do **not** flip any render sign
  (AD-037's load-bearing "rejected": a render flip double-inverts the jump).
- **Re-baseline geometry goldens** (box world-y changes deliberately, JC-017 style). The
  combat/advantage/determinism goldens must stay **unchanged** — the reflection is uniform,
  so hit/hurt *relationships* (and thus every combat outcome) are identical; if a combat
  golden moves, something is wrong (a non-uniform reflection), stop and raise it.

**Acceptance:** AD-037; `training-mode.md` criteria 5/14. Headless: a harness/test asserts
the sim-truth orientation now reads right-side-up (e.g. the crouch hurtbox's top edge is
*above* — more negative y — nothing, but its extent shrinks toward the feet; the pushbox
occupies the body's lower — nearer-to-feet — portion of the hurtbox). **Pixel visibility /
upright rendering is confirmed at the human-inspection re-gate** — not headless-checkable
(the whole point). **Judgment-log:** record any box whose "correct" reflected height is a
judgment (e.g. a spawn offset) for ratification.

### TKT-P1.1R-03 · Held-input stances — walk exit + crouch stance/block (AD-038)
**Serves:** **AD-038**; AD-032 (pure-direction command — the entry half); `combat-resolution.md`
phase 2; `character-a.md` → Movement ("standing and crouching blocking"). **Depends:**
TKT-P1.1R-01 (verify through it). Carries the only engine change in this file.
**Scope:**

- **Engine (AD-038):** in `step_phases.gd` `phase2_state_machine`, the actionable branch —
  when `Actionability.is_actionable(p, move)` **and** `move.loop`, re-derive the desired
  state each tick: `target = _buffered_command(...)`; if `target == -1`, `target =
  character.idle_state_id`; transition iff `target != p.state_id`. Non-loop actionable states
  keep the existing behavior. This gives walk **and** crouch their release→idle exit. Keep it
  character-agnostic (reads `move.loop` + `idle_state_id`, no character-A branch).
- **Content:** add the bare-`DOWN` pure-direction `button_map` entry `_map(-1, DOWN, 0,
  STATE_CROUCH)` so held `2` enters the existing `CROUCH` state. **Ordering** (AD-032
  first-match-wins): after the `DOWN+button` crouch normals (so `2L/2M/2H` still win when a
  button is held) and **before** the walk entries (so a down-forward `3` crouches, not walks).
- **Crouch block falls out** (no new mechanism): `_defender_is_blocking` already blocks on a
  held-back input in any non-hitstun actionable state, so once the crouch **stance** is
  reachable, a crouching defender holding down-back blocks (stand block already works). Note:
  blocking is **stance-agnostic hold-back** in the slice (no high/low enforcement) — that is
  the current model, not a gap this ticket introduces; a high/low block system, if ever
  wanted, is a Strategist scope call (do **not** add it here).

**Acceptance:** AD-038; `character-a.md` Movement. Harness assertions (brief checklist):
walk forward enters on held `6` and returns to idle on release; walk back likewise on `4`;
crouch enters on held `2` (crouch hurtbox resolves) and returns to stand on release; a
crouching held-back defender blocks a hit (enters a blockstun category). Determinism
unchanged (pure function of history + state); movement goldens change deliberately (walk now
terminates). **Judgment-log:** record any latitude (crouch entry ordering, block-state edge
cases) for ratification.

### TKT-P1.1R-04 · Airborne actions — directional/diagonal jumps + air normals (AD-039)
**Serves:** **AD-039**; AD-032 (composite pure-direction command); AD-015 (`CancelRule`);
`character-a.md` → Movement ("jumping neutral/forward/back … jump-in normals"). **Depends:**
TKT-P1.1R-01 (verify), and cleanest after TKT-P1.1R-03 (shares `content/character_a.gd`
`button_map`). Pure content — no engine/format change.
**Scope:**

- **Directional/diagonal jumps:** author `PREJUMP_F`/`PREJUMP_B` mirroring the existing
  `PREJUMP` (each an `input = 0` ALWAYS cancel, window `[3,3]` per JC-038, into `JUMP_F`/
  `JUMP_B` respectively). Add composite-direction `button_map` entries: `UP|FORWARD →
  PREJUMP_F`, `UP|BACK → PREJUMP_B`, and keep `UP → PREJUMP_N` — the **two diagonals listed
  before** bare `UP` (first-match-wins). `9` (up+forward) = forward jump; `7` (up+back) = back
  jump — the same motion the brief calls "forward/back" and the checklist calls "diagonal."
- **Air normals:** add three `CancelRule`s to each of `JUMP_N/F/B` — condition `ALWAYS`,
  `window` = the airborne frames (e.g. `[1, JUMP_DURATION − 1]`), `input =` `BUTTON_0/1/2`
  bitmask → `STATE_JL/STATE_JM/STATE_JH`. No `button_map` entry for the air normals is needed
  (`CancelEval`'s raw-button fallback resolves `rule.input == BUTTON_n`).
- **Known limitation (AD-036 class, do NOT build around it):** an air normal ends → idle at
  whatever height the arc/normal left it (no ground-contact landing — AD-036 deferred).
  Reachability + rendering is the bar; if the post-air-normal float reads badly at the gate it
  is an AD-036 roadmap question (Strategist), not this ticket.

**Acceptance:** AD-039; `character-a.md` Movement. Harness assertions: `9`→`PREJUMP_F`→
`JUMP_F` (pos_x carries forward), `7`→`PREJUMP_B`→`JUMP_B` (carries back), `8`→`JUMP_N`
(neutral, arc nets zero — JC-047); a button pressed mid-jump enters the matching `j.*`
(category AIRBORNE, reachable). Neutral-jump arc still lands flush (pos_y returns to
`ground_y`). **Judgment-log:** record any latitude (air-normal cancel window bounds, prejump
authoring) for ratification.
