## Sequencing (per-ticket dispatch — the P1.1/P2 default)

Per-ticket dispatch (protocol § "Token economy"): **one ticket per Developer
subagent, in dependency order, each ending on a real checkpoint.** No batching —
these three are small, mostly independent, and each lands on its own checkpoint;
there is no high spec-read overlap that would earn a cluster.

```
01 geometry-visible ─▶ 02 control-surface        03 serialization-version
   (player-init +          (binds controls onto      (AD-034; independent —
    AD-035 framing)         01's real players)         dispatch any time)
```

Dispatch order and why:

1. **TKT-P1.1-01 (geometry visible)** first. It carries the **player-init fix** —
   both players must start as the installed character in idle (character A, id 2,
   idle state 100), which the P1 shell never did (it left them as `new_initial()`'s
   generic `character_id 0 / state_id 0`, so the roster keyed on id 2 resolves
   nothing → empty boxes and zero readouts — the "state 0 / startup 0" the human saw).
   Every other readout and TKT-P1.1-02's operability depend on real players existing,
   so this lands first. *Checkpoint:* both characters' boxes visible on screen at rest.
2. **TKT-P1.1-02 (control surface)** second. Binds human controls onto the existing
   control methods and completes the device sampler (attack buttons). Depends on 01
   for real players to drive and observe. *Checkpoint:* a human can pause, frame-step,
   reset, and record/playback from bound controls, and perform character A's moves.
3. **TKT-P1.1-03 (serialization version)** is **independent** of both — a localized
   `sim_state.gd` change, no training-mode dependency. Dispatch it any time (last is
   fine); it does not gate the human gate.

**Geometry ruling (per the task): it is BOTH.** The blank overlay has two distinct
causes — a **pure code defect** (players never wired to the installed character, so
no boxes resolve) *and* an **unspecced render-framing contract** (no world→viewport
transform, so even resolved boxes fall off-screen / behind the panels at
`PX_PER_UNIT = 1` with world origin at the viewport top-left). The framing contract
is the Architect's — recorded as **AD-035**. The code defect is the Developer's.
Both are needed to make boxes visible, and neither is verifiable without the other,
so they land together in **TKT-P1.1-01** (fix player-init first *within* the ticket,
then framing, then confirm). The Developer's geometry flag (`flags.md`, owner:
Developer, 2026-07-08) closes when this ticket lands — its "may bounce to the
Architect if projection/framing is unspecced" clause is answered by AD-035.

**Human gate:** after 01 and 02 land (03 is orthogonal), the user runs
`training_mode.tscn`, operates every control, and confirms both characters' boxes
render live. Only then is P1 done (roadmap P1.1 done-bar). QA re-folds the visual
outcome into `audit-p1-feature.md`.

**Scope guard:** these tickets are bounded to the brief's required outcomes
(operability + visible geometry). **No new readouts, no training-suite features**
(combo trials, multi-slot recording, dummy AI, frame-data UI polish) — flag if any
creep in (`training-mode.md` → Scope boundaries).

### TKT-P1.1-01 · Geometry visible (player-init defect + render framing)
**Serves:** `training-mode.md` → "Geometry framing" + criteria 5, 14; **AD-035**
(render framing); AD-024 (players resolve the installed character); AD-019 (px scale
it composes with). **Depends:** the P1 training mode (shell, geometry overlay, the
inspection surface) — all landed. No dependency on 02/03.
**Scope — two parts, in this order:**

- **Part A — player-init code defect (Developer's own; pure code).** In the
  `TrainingMode` shell (`game/scenes/training_mode.gd`), both players must begin as
  the **installed character in its idle state**, not the generic default
  `SimState.new_initial()` builds. Today the shell installs character A under
  `CharacterA.CHAR_ID` (= 2) but the two players keep `character_id 0 / state_id 0`,
  so `PlayerView`'s `roster.has(character_id)` is false → `move == null` → `boxes ==
  []` and every frame-data/state readout reads zero (the 2026-07-08 finding). Set
  both players' `character_id` to the installed id and their `state_id` to that
  character's idle state (`Character.idle_state_id`, = `STATE_IDLE` 100 for A) as
  part of building the initial state. Keep it **character-agnostic** (read the id and
  idle state from what `configure()`/`_install_roster()` installed — no
  character-A-specific branch beyond the default the shell already picks). This is a
  wiring completion, not a new contract — no AD, no `SimState` shape change.
- **Part B — render framing (implements AD-035).** Add a **render-only** world→screen
  framing so both characters' resolved boxes are **fully visible within the viewport
  and unoccluded by the readout panels**. A `Camera2D` on the world/`GeometryOverlay`
  layer is the natural mechanism (frame the play area `wall_left..wall_right` /
  `ground_y` centered, ground seated low, zoom-to-fit with margin). The four readout
  `Control` panels stay **screen-anchored HUD** — *not* moved by the camera (so they
  stay put while the world frames behind/around them; this is what resolves the
  "boxes behind the x≈16–700 panel region" symptom). The transform is float,
  render-side, reads sim truth only to frame, writes nothing back — **never enters a
  snapshot or the canonical hash** (Tenet 1 intact, exactly as AD-019). Exact zoom /
  anchor / ground-line-y are **placeholder** (Developer's, like tuning); the contract
  is "both characters fully visible, unoccluded, world-layer separate from HUD-layer."

**Acceptance:** `training-mode.md` criteria 5 and 14. Headless: the geometry
view-model still produces the correct per-box draw entries (existing
`test_geometry_overlay.gd` stays green), and a golden taken with vs. without the
camera is identical (AD-019 criterion 6 / AD-035 — the framing is render-only, no
snapshot field). A test should also show that with the shell's initial state, both
players resolve a non-empty `boxes` list (Part A regression guard: `character_id`
matches the installed roster, idle move resolves its hurtbox). **Pixel visibility
(boxes actually on screen, unoccluded) is confirmed at the human-inspection gate** —
it is not headless-checkable, which is the whole point of the gate.
**Judgment-log:** record any latitude (e.g. exact camera params, whether a
`Camera2D` vs. a root offset) for Architect ratification before audit.

### TKT-P1.1-02 · Human control surface (bind controls + device buttons + legend)
**Serves:** `training-mode.md` → "Human control surface (operability — P1.1)" +
criterion 13; AD-018 (three attack buttons at the input layer). **Depends:**
**TKT-P1.1-01** (real players to drive/observe) and the P1 control methods already on
the shell (`set_paused`/`is_paused`/`step_once`/`capture_reset`/`do_reset`/
`set_dummy_mode`/`get_dummy_mode`, all landed).
**Scope:**

- **Bind each control operation to a device/keyboard control**, routed **through the
  `TrainingMode` shell's control methods** — never bypassing the shell into
  `TickHost`/`TrainingHarness`/`RecordPlaybackSource` directly (the shell stays the
  one place the sim is driven from, mirroring the read-only rule on the inspection
  side, criterion 10). Cover: pause/resume, frame-step (`step_once`), capture-reset,
  do-reset, and dummy record/playback mode-switch. A control-input handler
  (`_unhandled_input` / an input-map action set) is the natural home — keep it in the
  shell.
- **Complete the P1 device sampler.** `_sample_device_p1` currently reads directions
  only; also sample the **three attack buttons** (`BUTTON_0/1/2`, AD-018) into the
  emitted `InputFrame`, so a human can perform character A's moves and read their
  frame data. Still the one `InputSource` interface (Tenet 2) — the sampler emits the
  identical raw frame; no "P1 is special" type.
- **Surface a minimal on-screen controls legend** (a `Label`/panel listing the bound
  keys) so a human can discover the controls without reading code. Legibility of the
  instrument, **kept minimal** — not UI polish.
- **Key/action choice is placeholder** (Developer's, like tuning); default input-map
  actions live in `project.godot`. The contract is "each operation reachable from a
  bound control; controls discoverable."

**Acceptance:** `training-mode.md` criterion 13 (and it makes criteria 1–4, 7, 9 —
already headless-green via direct method calls — *exercisable by a human in-mode*).
Headless: a test can drive the bound-control handler (simulate the input action) and
assert it calls the corresponding shell control method / that the device sampler
encodes the three attack-button bits. **Operability itself is confirmed at the
human-inspection gate** (a human pressing real controls is not headless-confirmable).
Determinism unchanged — pausing/stepping never rewrite state (AD-004/AD-010); the
device source stays a pure `InputSource`. **Judgment-log:** record the chosen keys /
input-map actions and any latitude for ratification.

### TKT-P1.1-03 · Serialization format-version field (AD-034)
**Serves:** **AD-034**; `simulation.md` → Serialization; Tenet 1. **Depends:**
nothing in P1.1 — a localized `game/sim/sim_state.gd` change. **Independent; dispatch
any time.**
**Scope:**

- Add `const FORMAT_VERSION := 1` on `SimState` and emit `"v": FORMAT_VERSION` on the
  **top-level** `SimState.to_dict()` only (no sub-object dict gains a version — one
  version governs the whole graph, AD-034).
- In `SimState.from_dict`, read `var v := int(d.get("v", 1))`: **absent ⇒ 1** (legacy,
  the current shape — parse normally); **== FORMAT_VERSION ⇒ parse normally**;
  **anything else ⇒ fail loudly** (`push_error` with a clear message naming the
  unexpected version; do not silently proceed). No migration branch yet — there is
  one version; the older-version up-convert branch is added when v2 exists (a revision
  to AD-034).
- **Do not fold `"v"` into `hash_state()`** — it is format metadata, not mutable sim
  truth (AD-034; same exclusion as the install-generation token AD-024 and px
  projections AD-019). Adding the field must change **no** existing state hash.

**Acceptance:** `to_dict`/`from_dict` round-trip stays exact (the `"v"` field survives
the trip); `hash_state()` is unchanged for every existing state (a state's hash before
and after this change is identical — the field is excluded); a dict **without** `"v"`
still restores (absent ⇒ 1); a dict with an unrecognized `"v"` (e.g. 2) triggers the
fail-fast guard rather than a silent mis-parse. `simulation.md` determinism/round-trip
criteria stay green. **Judgment-log:** record any latitude (e.g. exact error text,
where the guard lives) for ratification.
