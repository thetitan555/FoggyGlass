# Comparative Review — FoggyGlass vs. Prima AI Infrastructure

*Written 2026-07-05 by the Prima Strategist (Claude Fable), acting as an outside senior reviewer of both systems. Basis: for FoggyGlass, the two provided documents only — "The FoggyGlass Pipeline — a portable description" and "Strategic Review — FoggyGlass (Fable, 2026-07-05)"; the repo itself was not available, so FoggyGlass claims are taken as reported and labeled where that matters. For Prima, the full live canon was read today (LIVE-RULES v4.9, ROLES, ARCHITECTURE, ENGINE-CATALOG, SYSTEM-MAP, all five engine specs, the routing playbook, the 2026-07-04 pre-go-live review, and the live PRIMA - Tasks sheet), which is authoritative over the portable summary. This document is advisory input for both projects — nothing in it is a decision, and nothing in either system was modified to produce it.*

---

## 1. VERDICT UP FRONT

**FoggyGlass** got the governance layer right and can prove it: one-owner-per-artifact, upstream correction, the judgment log with Architect ratification, and evidence-backed audits all demonstrably fired under load — contract gaps were flagged instead of hacked around, and QA caught a vacuously-green test. Its file-tree-as-memory model is the cleanest possible fit for a closed, reversible, git-backed world. Its single biggest risk is stated plainly in its own review: two milestones of verification against the spec, zero verification against the actual subject of the charter — a human player. The pipeline can currently prove the game matches the documents, not that the documents match reality.

**Prima** got the reality-contact problem right: external systems of truth that outrank every internal file, verification as a discipline binding every role rather than a stage, hard human gates on the irreversible (money, contracts, client sends), and monitoring that checks *outcomes* (did the output actually appear, is it fresh) rather than process (did the job run). It has survived real failure modes — stale state, lead-leak, a plugin that twice took down the live website — and encoded each as a rule. Its single biggest risk is that its behavioral guards live overwhelmingly in prompt text: the 2026-07-04 pre-go-live review found the "never overwrite Kim" rule enforced nowhere in the apply layer, and a dedup instruction that failed its first live test — instructed-against, not impossible, exactly the failure class FoggyGlass already fixed structurally.

## 2. CONVERGENCES

Built independently, in different domains, the two systems arrived at the same core, and the convergence is itself evidence that these are properties of the problem (memory-less agents doing multi-week work), not stylistic choices:

Files are the memory and unwritten decisions don't exist ("if it isn't saved, it didn't happen" ≈ "a decision doesn't exist until written to a file, email, or calendar entry"). A small set of roles separated by exclusive write-ownership, not by skill ("one owner per artifact" ≈ non-overlapping write-scopes) — both even landed on four roles with a Strategist and an Architect doing near-identical jobs. The human deliberately holds only the irreversible gates (git push and play-feel ≈ money, contracts, sends, and pricing). "Done" requires evidence, not builder belief ("nothing is done until it clears audit" ≈ "'done' is not 'I believe it's done' — state what you checked"). The dominant cost is each session re-reading its binding inputs, so both keep a tiny always-read pointer file (CLAUDE.md ≈ the LIVE-RULES bootstrap) and both discovered the binding read grows without bound unless something prunes it. And both independently learned the same hard lesson: memory/summary layers must never be treated as current status — FoggyGlass as a founding rule, Prima via a three-strike stale-state postmortem that had to be canonized on 2026-07-04.

## 3. KEY DIVERGENCES

**Source of truth.** FoggyGlass: the repo *is* ground truth — closed world, git-reversible, nothing outside it. Prima: truth lives *outside* the system (Gmail, Drive, Calendar, QuickBooks), decays continuously, and outranks any internal file. Each is correct for its world; each would be wrong in the other's.

**Shape of work.** FoggyGlass is a linear pipeline (idea → brief → spec → code → audit → done) with a terminal quality gate. Prima is a standing operation — concurrent event-driven loops (inbox sweeps, reconcilers, gap-fill, nightly audits) with no terminal state. A pipeline can afford a blocking gate; an operation whose billing rhythm is "invoice the same day the milestone passes" cannot.

**Verification model.** FoggyGlass concentrates verification in a dedicated QA role with a written audit criterion and pass/fail authority on objective failures. Prima distributes it: a discipline binding every role ("verify against a source of truth and cite it, or mark it unverified") plus scheduled machine monitors (nightly SYSTEM-MAP reconcile, the closed-ledger check, the 18-row Outcome Ledger that checks every external seam's *output* freshness). Domain-fit on both sides: in a build, "done" is a discrete event you can gate; in operations, truth rots ambiently, so verification must be ambient too.

**Concurrency.** FoggyGlass serializes everything through the human as the relay bus (deliberately, to learn the system; its review says this debt is now due). Prima runs true concurrency — two humans and multiple agent surfaces co-editing the same Google Sheets — and needed real machinery for it: the _queue/apply mechanism, an ID-collision guard, claim-before-work, and write-back-at-completion.

**Formalization of judgment.** FoggyGlass logs every builder latitude call in one running judgment log, and the Architect ratifies or overturns each before audit — provisional decisions never silently become permanent. Prima has a partial analog (the routing playbook's learn loop: a triage answer becomes a dated LEARNED RULE) but no general judgment log; automations' embedded judgment calls accrete in SKILL prompt bodies and get caught, if at all, by reviews after they're live.

**Cost model.** FoggyGlass measures spend per milestone in dollars and tiers models per role in one frontmatter line. Prima right-sizes models ad hoc (noted per task, e.g. Sonnet for the photo log) and its binding constraint is owner attention, not tokens — a defensible difference in emphasis, not a defect.

## 4. RECOMMENDATIONS FOR PRIMA (ranked)

**P-1. Enforce "never overwrite Kim" in the apply layer, not in prompts.**
(a) Adopt: structural constraint over instruction — make the forbidden write *impossible*. (b) FoggyGlass mechanism: "constrain roles with tool permissions, not just prompt text" — after its 150k-token QA runaway, the fix was removing the spawn tool from role frontmatter, turning the failure class off. (c) Benefit against owner attention: this is the highest-value transfer available. Prima's own 2026-07-04 pre-go-live review found the blank-only/never-overwrite rule exists only in task prompts (B1), that `applyOne_` in `apply_queue_to_sheets-GUARDED.gs` sets cell values unconditionally (H-a), and that a prompt-level dedup guard failed its first live test the same night (H-n). Every silent overwrite on Kim's Selections sheet costs exactly the thing the whole system exists to protect — her trust and her time untangling it. (d) Cost/risk: one Apps Script change (an `onlyIfBlank` / expected-old-value mode on `set` ops, refused-and-reported on conflict) plus updating requesters; low regression risk, testable on a dummy sheet. (e) Domain-fit: survives fully — Prima already has the single choke-point (applyQueue) where FoggyGlass had frontmatter; enforcement at a choke-point is *more* natural in Prima's architecture, not less.

**P-2. Adopt the judgment log + ratification loop for unattended automations.**
(a) Adopt: one running file where every automation and role session logs its latitude calls — uncertain matches it acted on, mappings it invented, thresholds it chose — logged at the moment of the call, never blocking; the Architect reads and ratifies-or-overturns on a fixed cadence (weekly audit is the natural slot). (b) Mechanism: FoggyGlass load-bearing idea #5, verbatim. (c) Benefit: Prima's worst near-miss class is precisely unratified latitude — the gap-fill's merged router quietly deciding "chosen/selected → Approved=Yes, Decided by=Client" was an agent-invented rule nobody ratified, caught only by a manual review three days before an unattended three-week window. The routing playbook's learn loop proves the pattern already works in one corner; generalize it. (d) Cost: one shared log file + one appended duty per task prompt + one Architect cadence duty; the read stays cheap if entries are one-liners. (e) Domain-fit: yes — recording-never-blocks matches Prima's silent-by-default rules, and ratification maps onto an existing Architect duty (folding surfaced decisions into canon deliberately).

**P-3. Numbered acceptance criteria for every automation change; audits cite them.**
(a) Adopt: each new or changed automation ships with numbered, checkable ACs, and its commissioning test verifies them by number. (b) Mechanism: FoggyGlass spec sections with numbered acceptance criteria + tickets citing exact sections + audits verifying against them (the P1 audit's 1131 real checks). (c) Benefit: Prima already does this instinctively at its best (the Engine-4 twelve-step acceptance test; the five-case matcher test) but not uniformly — the field-qa dedup "fix" of 7/3 had no written AC and failed silently on 7/4. Numbered ACs make "Scott validates one clean run" a checklist a fresh session can re-verify, which is what write-back-at-completion needs to be trustworthy. (d) Cost: modest doc weight per change. (e) Domain-fit: yes, with one adaptation — ACs attach to *changes and commissioning*, never to daily operations (no blocking gates on live work).

**P-4. A compaction cadence for the canon.**
(a) Adopt: a scheduled Architect pass that shrinks the binding read — superseded blocks collapse to one-line tombstones with a pointer, correction notes fold into the section they correct, status-update accretion rolls up. (b) Mechanism: FoggyGlass review item 4 — milestone-boundary compaction; superseded ADs become tombstones, ratified JCs prune to index lines. (c) Benefit: every Prima role re-reads canon every session; MARKETING-ENGINE now carries its own superseded 6/18 rules inline with 7/2 corrections of 6/24 sections, SALES-ENGINE carries five stacked status updates, and LIVE-RULES carries amendments-to-amendments. That's token cost on every session, and — worse for Prima's constraint — contradiction surface (the M-p doc-drift class its own nightly keeps flagging). (d) Cost/risk: the real risk is deleting load-bearing history; Prima's corrections (e.g., "the _wpqueue pipeline was never built") are safety rails. Tombstone, never delete; keep the correction's one-line conclusion, archive the narrative. (e) Domain-fit: yes — Prima already has the applyDocQueue path and a version-bump rule; it lacks only the cadence. Monthly, or at each engine's commissioning close.

**P-5. Formalize model-tiering in SYSTEM-MAP and measure once.**
(a) Adopt: a model-tier column on the SYSTEM-MAP scheduled-task table plus one month of spend measurement. (b) Mechanism: FoggyGlass one-frontmatter-line-per-role tiering plus its "measure spend, then tune from data" step. (c) Benefit: real but secondary — Prima's constraint is attention, not tokens; still, the Strategist's proportionality arena (ROLES scope C) currently has no spend data to judge "earns its keep" with. (d) Cost: trivial. (e) Domain-fit: yes.

## 5. RECOMMENDATIONS FOR FOGGYGLASS (ranked)

**F-1. Declare the player a system of truth that outranks the spec.**
(a) Adopt: Prima's precedence rule, transplanted — "if a derived file disagrees with a system of truth, the derived file is wrong." For FoggyGlass: *if the played experience disagrees with the spec, the spec is wrong.* Make play-reports first-class artifacts (owned by the human, like the charter) that can overturn ADs through the existing flag mechanism. (b) Prima mechanism: systems-of-truth precedence — the single defining trait of Prima's design, the thing that keeps a document-perfect system honest against reality. (c) Benefit against FoggyGlass's constraint (shipping something a human can feel): this converts its own review's item 1 ("the design has never met a human") from a scheduling fix into a structural one. A play gate added as a milestone step can slip; a declared truth-source with overturn authority cannot be argued away by a clean audit. Ten minutes of play generating brief-quality evidence is the cheapest tokens the project will ever spend. (d) Cost/risk: near zero — one protocol paragraph plus a docs/play-reports/ owner line. Risk: over-rotating to anecdote; bound it the way Prima bounds it (a truth-source is *evidence to reconcile against*, not an instruction to obey blindly — Prima still flags conflicts rather than silently rewriting). (e) Domain-fit: survives — this is the one place FoggyGlass's world is *not* closed: the player was always the external reality; the pipeline just hasn't been pointed at it.

**F-2. Claim-before-work + write-back-at-completion when per-ticket dispatch lands.**
(a) Adopt: before the orchestrator dispatches a ticket, the ticket row is claimed (status + session note); the moment a ticket completes, its status flips in the same turn — never at session close. (b) Prima mechanism: the 2026-07-04 Strategist ruling amendments — claim-before-work under concurrency, write-back at completion because "sessions get abandoned mid-flight; a close-time duty inherits the same staleness it's meant to fix." Prima learned this from three mid-session deaths' worth of stale state; FoggyGlass lost work to exactly three mid-session deaths in P1. (c) Benefit: per-ticket dispatch (its own review item 2) is a mild concurrency model — the first time two dispatches overlap or an orchestrator retries, unclaimed tickets get double-built and dead sessions leave ghost "in-progress" work. This closes both failure modes before they're experienced rather than after. (d) Cost: two lines in the protocol + one status field per ticket. (e) Domain-fit: yes — it's the minimum concurrency hygiene, and commit-as-you-go (already planned) is the same instinct applied to code.

**F-3. One open-items ledger across all artifact types.**
(a) Adopt: a single "open work" view — tickets, open flags, unratified judgment calls, unresolved audit findings — one place a fresh session reads to know the whole project's live state. (b) Prima mechanism: "PRIMA - Tasks is the ONE status source for open work of every kind, human and system; a memo may hold detail but is never the only place a to-do lives" — plus the Next-5 rule (any real item beyond the visible few must exist as a row, so item 6+ can never be forgotten). (c) Benefit: FoggyGlass currently splits live state across flags.md, tickets/, and the judgment log's unratified tail; its own review notes the judgment log is the one large file without a cheap-read front door. An index-ledger keeps the always-read set small (its stated cost doctrine) while guaranteeing nothing lives only in a big file. (d) Cost: one file + a discipline; the risk is double-entry drift, which Prima solves by making the ledger an *index* (pointers), never a copy. (e) Domain-fit: yes.

**F-4. The "do the mechanical work" rule — retire the human bus now.**
(a) Adopt: Prima's rule verbatim — "never hand a human work a computer does faster; human time is reserved for decisions, approvals, and the gates." (b) Prima mechanism: LIVE-RULES "Do the mechanical work"; in Prima the human relay was never allowed to exist. (c) Benefit: FoggyGlass's review already concludes the user-as-bus deferral is due (item 5); this is the principled version of that call — the human stays on exactly two gates (push, play-and-report) and stops being the serialization bottleneck. Directly serves the token/dollar constraint too: human relay latency is what motivated the failed big-batch doctrine in the first place. (d) Cost: the orchestrator build it already plans. (e) Domain-fit: yes, *with the training-wheels caveat honored* — its protocol kept the bus manual to learn the system; one milestone has now run, so the learning purpose is served.

**F-5. An outcome check on the automated dispatcher (small).**
(a) Adopt: when orchestration is automated, verify the *artifact landed* after each dispatch (file exists, tests ran, commit present), not that the sub-session ran. (b) Prima mechanism: the Outcome Ledger — "health checks measure that a duty RAN, not that its output WORKED"; check output existence + freshness. (c) Benefit: modest but real — an orchestrator that believes a dead sub-agent succeeded reintroduces the stale-state class silently. (d) Cost: a few lines in the orchestrator. (e) Domain-fit: partial — a closed pipeline usually fails loudly, so a full seam-monitoring table would be dead weight (see §6); the thin per-dispatch check is the right-sized transplant.

## 6. WHAT EACH SHOULD NOT ADOPT

**Prima should not adopt:**
- **The linear pipeline / terminal audit gate.** Operations are loops against decaying truth, not a line with an end. A blocking "nothing is done until it clears audit" gate on live work would delay the same-day invoice and the lead reply — directly attacking the business outcome. Prima's ambient verification + scheduled monitors is the correct inversion. (Gates belong only on *changes to automations*, which P-3 covers.)
- **A dedicated QA role.** A fifth role is governance overhead priced in owner attention; Prima's own canon warns the roles are scaffolding, not where value is created. Verification-as-discipline plus the nightly/weekly machine audits already provide the function without another surface to maintain.
- **Builder latitude as a broad grant.** FoggyGlass gives its Developer real freedom over "how" because git makes every mistake reversible. Prima's world is irreversible at the edges (sends, money, Kim's cells); its near-zero-latitude posture there is load-bearing. Adopt the judgment *log* (P-2), not the latitude.
- **Human-as-relay-bus, even temporarily.** Fails Prima's own "do the mechanical work" rule; owner attention is the binding constraint.

**FoggyGlass should not adopt:**
- **External systems of truth (beyond the player, F-1).** The repo is ground truth in a closed world; inventing external authorities adds reconciliation cost with nothing to reconcile against.
- **Propose-not-execute human approval on routine changes.** Prima needs it because actions are irreversible; git reversibility is precisely what makes autonomous execution safe inside the pipeline. Keep humans on two gates, not in every loop — otherwise the token/dollar constraint loses to approval latency.
- **The _queue/apply mechanism.** It exists because humans and agents co-edit one live spreadsheet Claude can't write directly. FoggyGlass's one-owner-per-file rule is the stronger, simpler solution and already holds.
- **Standing watchdogs, digests, and seam monitors at Prima scale.** Prima's operation decays silently 24/7 with owners abroad; a pipeline between sessions is inert. Milestone drift sweeps + the thin F-5 check are the right amount; a daily digest of a dormant repo is busy work — which Prima's own no-busy-work rule would reject.
- **Silent-by-default.** That is a Kim-protection rule for a co-owner's working surfaces. QA should stay loud.
